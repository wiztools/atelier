# fal.ai Image Generation Integration

## Summary
Add fal.ai as a **selectable image-generation backend** alongside the existing Ollama image path. fal.ai uses the **queue API** (`queue.fal.run`): submit → poll status → fetch result. Both backends remain functional; the user picks via a new `config.Models.ImageProvider` setting (`"ollama"` default, `"fal"` opt-in). Default fal model: `fal-ai/flux/schnell`.

The integration slots into the existing `generate_image` tool seam — **no changes to the harness planning loop, triage, image stripping, history persistence, or UI rendering**. The provider dispatch happens in one place (`tool_gateway.go`).

## Design decisions
- **fal.ai is NOT a ChatProvider.** It doesn't fit `provider.go`'s `ChatProvider` interface (no chat/list-models streaming). It only generates images. So it gets a single new file `fal_client.go` — no provider adapter, no registry entry. This avoids forcing fal into the chat-provider shape.
- **fal client returns the same shape the tool already harvests.** The tool's `Execute` harvests images from `ollamaGenerateResponse` + raw JSON. The fal client builds a synthetic `ollamaGenerateResponse` (with the image as a base64 data URL) and returns the raw fal JSON. Existing harvest logic (`normalizeImagePayloads`, `collectImagesFromJSON`) works unchanged.
- **fal client fetches the result image URL → base64 itself.** fal returns image *URLs*; `decodeImagePayload` (used to write artifacts to disk) only handles bytes, not URLs. So the fal client downloads the image and embeds it as a data URL before returning. This keeps the rest of the pipeline (which assumes base64) untouched.
- **Caption model logic unchanged.** `responseModelFor`/`responseProviderFor` force the Ollama harness text model to write the caption in image mode — still correct regardless of which backend produced the image. No change there.

## Files to create

### `fal_client.go` (new)
Mirror `openrouter_client.go` style:
- `const falQueueBaseURL = "https://queue.fal.run"`
- `type FalClient struct { httpClient *http.Client; apiKey string }`
- `newFalClient(httpClient *http.Client, apiKey string) FalClient` — trims key, falls back to `http.DefaultClient`.
- `GenerateImage(ctx, req ImageGenerateRequest) (ollamaGenerateResponse, []byte, error)` — the full queue flow:
  1. Resolve model: `req.Model` or fall back to `defaultFalImageModel = "fal-ai/flux/schnell"`.
  2. **Submit**: `POST {base}/{model}` with `Authorization: Key {apiKey}`, body `{"prompt": ..., "image_size": {"width":W,"height":H}, "num_inference_steps": steps, "num_images": 1}` (translate from `ImageGenerateRequest`). Parse `request_id`.
  3. **Poll**: `GET {base}/{model}/requests/{id}/status` every 1.5s until `status == "COMPLETED"` (or `IN_QUEUE`/`IN_PROGRESS` continue), with an overall deadline (3 min via context). Surface `COMPLETED`/failed/error states as typed errors.
  4. **Fetch result**: `GET {base}/{model}/requests/{id}` → extract `images[].url`.
  5. **Download** each URL → decode → re-encode as `data:image/<ext>;base64,...`.
  6. Return synthetic `ollamaGenerateResponse{Model, Image: <first dataURL>, Images: [...]}` + raw fal JSON.
- Private `do(ctx, method, path, body)` helper mirroring OpenRouter's: key guard (empty → `"fal api key is not configured"`), `Authorization: Key`, 401/429 typed messages, 4096-byte error body read.
- Wire-format structs: `falSubmitResponse{RequestID}`, `falStatusResponse{Status, Logs, Error}`, `falResultResponse{Data json.RawMessage, Images []falImage}`.

### `fal_client_test.go` (new)
Table-driven tests using the `roundTripFunc` mock pattern from `app_test.go`:
- Happy path: submit → two polls (IN_PROGRESS then COMPLETED) → result with image URL → base64 output.
- Submit error (401, invalid key).
- Poll timeout (status stuck → context deadline).
- Result with multiple images.
- Model fallback (empty `req.Model` → default).

### `tool_gateway.go`
Rewrite the `GenerateImage` closure to dispatch on `config.Models.ImageProvider`:
```go
gateway.tools.GenerateImage = func(ctx context.Context, req ImageGenerateRequest) (ollamaGenerateResponse, []byte, error) {
    if strings.TrimSpace(config.Models.ImageProvider) == "fal" {
        apiKey, err := loadFalAPIKey()
        if err != nil { return ollamaGenerateResponse{}, nil, err }
        if apiKey == "" { return ..., errFalKeyNotConfigured }
        return newFalClient(app.client, apiKey).GenerateImage(ctx, req)
    }
    return app.ollamaClient(config.Providers.Ollama.BaseURL).GenerateImage(ctx, req)
}
```
Add `errFalKeyNotConfigured = errors.New("fal api key is not configured")` (mirrors `errOpenRouterKeyNotConfigured`).

- **Registration gate** (line 64-70): register `generate_image` if *either* provider is configured — add helper `imageGenerationConfigured(config AppConfig) bool` (Ollama image model set, or `imageProvider == "fal"` with a model/key present).

## What stays untouched (verified)
- Triage / `responseMode="image"` forcing (`triage.go`, `harness.go:168-172`).
- Image stripping from model context (`harness.go:971-978`).
- `imagesFromToolResults`, `writeChatImageArtifacts`, `appendChatAssistantTurnWithImages`, `hydrateHistoryContent`, `/atelier-artifact/` URL handling.
- `responseModelFor`/`responseProviderFor` (caption always = Ollama harness text model).
- Image rendering in the UI (`chat-image-results` grid, download, history hydration).
- Telemetry redaction (`compactRawResponse`/`redactLargeImageStrings`) — provider-agnostic.

## Build & test commands
- `gofmt -w *.go` (per `bld.sh` convention — root-level glob).
- `go test ./...` — run all Go tests (new fal tests + existing).
- `npm run build --prefix frontend` — type-check + build frontend.
- `wails build` only if you want a full app bundle (optional; `wails dev` regenerates bindings).

## Out of scope (intentionally)
- A fal model *catalog/list* endpoint — model is free-text with a default (simpler; fal has no clean catalog API like OpenRouter's `/models`).
- Webhook/streaming status — plain polling is sufficient for a desktop app.
- Migrating existing Ollama image-gen users — `ImageProvider` defaults to `"ollama"`, so current behavior is unchanged until the user opts in.
