# Plan: Hybrid native tool-calling for the Ollama planner

## Goal

Let the harness **planner** use Ollama's native function-calling API (`tools` request param + `message.tool_calls` response) when the harness model advertises a `tools` capability. Otherwise fall back to the existing grammar-constrained `format`-schema path unchanged. The **final/answering model never sends `tools`** (Invariant 1 preserved). Per your decisions: trigger = auto-detect via capabilities; telemetry = derive `brief` from the planner's `content`.

## Core architectural decision: a fork inside `prepareChatTurnLoop`, not a parallel pipeline

`prepareChatTurnLoop` (`harness.go:609`) gains a `useNativeTools bool` parameter. Everything outside the per-round body is shared: skill selection, registry, context budgeting, the bounded loop (`harnessChatMaxSteps`/`harnessChatMaxWallTime`/≤3 calls), telemetry recording, tool execution via `ToolGateway`, and the result-feedback mechanism. Only **three things differ** per round:

| Concern | Format-schema path (existing) | Native path (new) |
|---|---|---|
| **Request** | `Format: harnessToolPlanSchema(...)`, no `tools` | `Tools: ollamaToolSpecs(registry)`, no `Format` |
| **Parse plan** | `parseHarnessToolPlanWithRegistry(content)` → `{brief, needsTools, reason, toolCalls}` | map `Message.ToolCalls` → `[]HarnessToolCall`; `brief = content`; `needsTools = len(calls) > 0` |
| **Feedback on invalid call** | `role:"user"` correction message (current, `harness.go:659-663`) | `role:"tool"` error message (native idiom) |

## File-by-file changes

### 1. `app.go` — request/response structs carry tool calls

- **`ChatRequest`** (`app.go:176`): add `Tools []map[string]any \`json:"tools,omitempty"\``. Populated only on planner calls; the final-response request (`preparedResponseRequest`, `harness.go:756`) leaves it nil.
- **`ollamaChatResponse`** (`app.go:236`) and **`ollamaChatChunk`** (`app.go:223`): add `ToolCalls []ollamaToolCall \`json:"tool_calls"\`` to their embedded `Message` struct.
- New type `ollamaToolCall { Function struct { Name string; Arguments json.RawMessage } \`json:"function"\` }` — mirrors Ollama's wire shape.
- **`ChatCompletionResult`** (`ollama_client.go:19`): add `ToolCalls []ollamaToolCall` so the parsed calls survive from `CompleteChat` into the loop.

### 2. `ollama_client.go` — send `tools`, decode `tool_calls`, detect capability

- **`OpenChatStream` / `CompleteChat`** (`ollama_client.go:175`, `:196`): when `req.Tools != nil`, set `body["tools"] = req.Tools`.
- **Capability detection**: add `hasToolsCapability(capabilities []string) bool` next to `hasImageGenerationCapability` (`ollama_client.go:129`), matching on `"tools"` (normalized). No new API surface — `ShowModel` (`ollama_client.go:108`) already returns `Capabilities`.

### 3. `tools_registry.go` — emit Ollama `tools` specs

- Add `ParamSchema map[string]any` field to `HarnessToolDefinition` (`tools_registry.go:19`), populated declaratively per tool (`list_files`, `read_file`, `run_command`, `write_file`, `generate_image`). This promotes the rules currently embedded procedurally in each `Validate` func into a JSON Schema. The `Validate` funcs stay as runtime backstops — the native path still calls them after mapping.
- Add `ollamaToolSpecs(registry HarnessToolRegistry) []map[string]any`: maps each definition to `{"type":"function","function":{"name","description","parameters"}}`. This is what goes into `body["tools"]`.

### 4. `harness.go` — the fork, fail-safe capability check, planner prompt variant

This is the largest change.

**(a) Capability resolution in `RunChatStream` (`harness.go:113`).** After triage decides `needsTools`, and only then, resolve tool support: call `h.app.ollamaClient(req.BaseURL).ShowModel(ctx, harnessModel)`; if it succeeds and `hasToolsCapability(show.Capabilities)` → `useNativeTools = true`. **Any error or absent capability → `useNativeTools = false` (format-schema path).** This is the fail-safe: native tools are an enhancement, never a requirement. The `ShowModel` call is local and fast; it runs once per tool-needing turn (not for plain text responses). Pass `useNativeTools` through `harnessTurnContext` (`harness.go:500`) into `prepareChatTurnLoop`.

**(b) The loop fork (`harness.go:620-686`).** Inside the `for iteration` loop, branch on `turn.UseNativeTools`:
- **Request build**: native → `Tools: ollamaToolSpecs(registry)`, no `Format`; format → `Format: harnessToolPlanSchema(registry)` (current code).
- **System prompt**: native path calls a new `plannerSystemPromptNative(...)` variant — keeps the role/skill/workspace-root/tool-task/"failed-is-information" guidance, drops the JSON-envelope description, and instead instructs: "You have tools. Call them for evidence. When done, write a one-line summary of your plan in your content and make no tool calls." Format path keeps `plannerSystemPrompt`.
- **Parse**: native → `mapNativeToolCalls(completion.ToolCalls)` returning `[]HarnessToolCall` (unmarshal each `arguments` JSON object directly onto a `HarnessToolCall` — the flat shape already matches); `brief = strings.TrimSpace(completion.Content)`; `needsTools = len(calls) > 0`. Format path keeps `parseHarnessToolPlanWithRegistry`.
- **Validation**: native path still runs `validateHarnessToolCall` per mapped call (reuse the existing per-tool validators). On failure, feed the error back as a `role:"tool"` message (`{"status":"failed","error":"..."}`) — native idiom — instead of the format path's `role:"user"` correction (`harness.go:659-663`).
- **Loop termination**: native → stop when `len(toolCalls) == 0` (no `needsTools` flag exists). Format → current `!plan.NeedsTools` check (`harness.go:668`).

**(c) Result feedback is shared.** Both paths append `assistant` content + tool-result messages and re-plan on evidence. `toolResultMessages` (`harness.go:810`) and `runHarnessToolCalls` (`harness.go:1058`) are unchanged.

**(d) `preparedResponseRequest` (`harness.go:756`) is unchanged** — it strips `Tools` (never copies it), so the final model stays tool-free. Invariant 1 holds for free.

**(e) `appendToolEvidenceToSystem` (`harness.go:788`) and `toolEvidenceUserMessage` (`harness.go:776`) are unchanged** — the final model still gets observations as a single `role:"user"` message, preserving the Mistral role-ordering fix (Invariant 2).

## What does NOT change (the four invariants)

1. **Primary model never calls tools** — `preparedResponseRequest` never sets `Tools`. ✓
2. **Cross-provider role-ordering** — only the Ollama planner uses native tools; the final model (possibly OpenRouter/Mistral) still gets `role:"user"` evidence. ✓
3. **Validation/correction** — native path still validates each call and feeds errors back (as `role:"tool"`); only the feedback channel changes, not the guarantee. The format path keeps its grammar + correction loop verbatim. ✓
4. **Telemetry separation** — `brief`/`reason` remain telemetry-only, never enter the primary model's prompt. `appendToolEvidenceToSystem` only emits the fixed code-authored notes. Native rounds populate `brief` from content; `formatHarnessPreparationThinking` (`harness.go:849`) renders it unchanged. ✓

## Test plan

All tests live in `app_test.go`, following the `roundTripFunc` transport-stub pattern (`app_test.go:22`).

**New unit tests (direct-function, mirror `TestHarnessToolPlanValidationUsesSuppliedRegistry` at `app_test.go:1259`):**
- `TestHasToolsCapability` — matches `hasImageGenerationCapability` pattern; asserts `"tools"`, `"Tools"`, absent, and image-only capabilities.
- `TestOllamaToolSpecs` — asserts each spec has `type:"function"`, correct `name`/`description`, and a `parameters` object; no spec for tools absent from the registry.
- `TestMapNativeToolCalls` — maps a canned `[]ollamaToolCall` (with `arguments` as a JSON object) to `[]HarnessToolCall`; covers `read_file`/`run_command`/`write_file`, an unknown tool name, and malformed `arguments`.

**New integration test (mirror `TestHarnessCanRequestSecondToolRound` at `app_test.go:2990`):**
- `TestHarnessPlansWithNativeToolCalls` — transport stub returns `/api/show` with `capabilities:["tools"]`; planner rounds return native `message.tool_calls` (round 1: `list_files`; round 2: `read_file notes.txt`; round 3: no tool calls + content = "I have the notes"). Asserts `prepCalls == 3`, two `tool_call` steps, the file content reaches the final model as `[Tool observations]`, and the saved run's round briefs equal the planner `content` strings.

**New fail-safe test (mirror `TestTriageChatTurnFailsSafeToToolPath` at `app_test.go:3530`):**
- `TestNativeToolsFallsBackWhenCapabilityAbsent` — `/api/show` returns no `tools` capability (or errors); asserts the planner uses the format-schema path (`payload["format"] != nil`, `payload["tools"] == nil`) and behavior is identical to today.

**Existing tests pass unchanged** because they don't stub `/api/show`, so `ShowModel` errors → `useNativeTools = false` → format-schema path. I'll verify with `go test ./...` before finishing.

## Out of scope (explicitly not doing)

- No native tool-calling for the final/answering model.
- No config flag (`ConfigOllamaModels.NativeTools`) — auto-detect only, per your decision.
- No changes to triage, skill selection, tool execution, permission gating, or history persistence.
- No caching of `/api/show` capabilities across turns (per-turn call; can add an in-memory cache later if profiling shows it matters).
